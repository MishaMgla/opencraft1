# client (web)

TypeScript source compiled in-place to ESM by `tsc` (no bundler). Emitted `.js` files live beside their `.ts` sources and are gitignored build artifacts. Run `cd web && npm run build` before serving locally (the Go server serves `web/` from disk). PixiJS v8 from CDN (the jsdelivr prebuilt ESM bundle, `https://cdn.jsdelivr.net/npm/pixi.js@8.19.0/dist/pixi.min.mjs` — esm.sh was found to break Pixi v8 extension registration during E2E testing). renders an isometric view of the shared world. The shipped client has **zero runtime npm dependencies**; devDeps are build-time only (`typescript`, `@types/node`, `pixi.js` types, `@vercel/node` types, Playwright).

## modules (`web/src/`)
- `iso.ts` — isometric projection (`worldToScreen`, `depth`). the only place that knows the camera is iso; the server is projection-agnostic.
- `wire.ts` — binary codec mirroring `internal/wire`, including movement, role choice on Hello, paint, ult activation, roster/ult state, shake frames, and cosmetic jump frames. **must stay byte-for-byte in sync with the Go side.**
- `net.ts` — WebSocket connect, send Hello/Input/Paint/Ult/Jump, dispatch decoded frames to handlers.
- `config.ts` — resolves the engine WebSocket URL: fetches `/config.json` (served by a Vercel function from `WS_URL` in the split deploy) and falls back to same-origin `ws(s)://${location.host}/ws` (scheme matched to the page) on any failure, so local single-process dev is zero-config.
- `input.ts` — keyboard state → local movement integration (client-authoritative), plus capture-phase `F` paint hold/one-shot state, one-shot `Space` cosmetic jump state, and one-shot `E` ult activation.
- `render.ts` — PixiJS app (loaded from the jsdelivr prebuilt ESM bundle): crisp non-antialiased retro-symbolic iso floor over the 8192×8192 playable world, shared painted tile overlays rendered as generated terrain graphics for the eight server paint colors (red lava, green grass, yellow sand, blue water, orange copper, purple crystal, cyan ice, pink flowers) with the old color diamond as deterministic fallback, generated horse character skins for players when present (facing selected from movement and matched to the four isometric diagonal facings `north-east`/`south-east`/`south-west`/`north-west`; falls back to nearest cardinal for legacy cardinal-only art). The renderer GROUNDS each skin itself: it auto-detects each sprite's feet row from the PNG alpha and drops the procedural token shadow, so a skinned character sits on its tile with no doubled shadow or hover. Manifest `animations.walk` frames are keyed by the same facing names and loop while moving; absent walk frames use the presentation-only fallback trot. Procedural player token fallback, local player name-label updates (lifted clear above the sprite), one-shot shake offsets, one-shot cosmetic jump offsets, depth-sorted by world position, camera follow, world-container zoom scaling.
- `main.ts` — orchestration: name/role entry or saved local username → connect → per-frame loop (move, interpolate remotes, center camera, rate-limited input send). persists the last username in browser `localStorage`, auto-joins with that username on reload, wires the generated `horse` skin for local and remote players when available, wires the top-left username button/profile modal for local username edits, wires HUD zoom buttons to renderer scale without touching page zoom, sends `F` hold paint requests once per newly entered paint tile, sends a one-shot cosmetic jump request on each new `Space` press, sends `E` ult activation, and renders the roster from server player-state frames while overriding the local row with the current local username. resolves the socket URL via `config.ts` (`resolveWsUrl()`) before connecting. exposes live `{me, others, bounds}` on `window.__game` when `window.__E2E` is set, for the Playwright smoke test (`web/e2e/`); inert otherwise.
- `index.html` — retro-styled name/role-entry overlay with concise issue-59 mechanics guidance (role choice, Pulse/Cross/Trail effects, normal-paint ult charge, `E` activation, hold-`F` painting, roster progress) + top-left HUD (clickable username/profile control, status, `?` controls help popover, viewport zoom buttons) + profile edit dialog + compact player roster HUD + fixed bottom-left GitHub repository link + module entry.
- `assets.ts` — loads sprite assets declared in `web/assets/manifest.json` (an additive registry generated by `web/tools/gen-asset.mjs` during `dev-implement`); falls back to procedural rendering when an entry is absent so the game runs without any generated assets.

## sharp edges
- remote players are smoothed toward the latest snapshot (`rx += (tx-rx)*0.2`); not time-based interpolation — good enough for MVP.
- input is sent at ~15 Hz as an absolute position; the render loop runs at display refresh.
- `Space` sends one cosmetic jump request for each new key press and never paints. `F` sends one paint request for each new key press; while held, it sends one more paint request each time the player enters a new rendered 128-unit tile. the server chooses the authoritative nearest tile center and decides whether charge should advance, so idle key repeat and same-tile requests do not farm ult charge. paint/shake/jump visibility comes back through server broadcasts rather than local-only prediction. the key handlers run in capture phase so focused HUD controls cannot consume the key first.
- `E` sends an ult activation request. readiness and progress are displayed from server `SPlayer` frames in the roster (`ready` or `n/12`), not predicted client-side.
- the username is anonymous and browser-local only. `main.ts` stores the last saved username under `opencraft1.username`; a saved value skips the initial overlay and joins with the existing default checked role. Editing the profile modal updates local HUD/roster/token presentation and future reloads, but it does not add accounts, cloud sync, or a mid-session server rename protocol.
- the welcome overlay should explain only the issue-59 role/roster/ult/hold-paint mechanics, not a full tutorial. keep its copy consistent with README guidance and the HUD controls help: Pulse paints a centered 3 x 3 block, Cross paints the current tile plus two tiles in each cardinal direction, Trail auto-paints the next 8 distinct entered tiles, normal painting charges ult progress, `E` activates ready ults, hold-`F` paints newly entered tiles, `Space` jumps, and the roster shows connected players with role and progress. When controls change, update the HUD `?` help copy and this doc together.
- HUD `+`/`-` controls scale the Pixi world container in 10% steps from 50% to 150%; this is renderer presentation only, not browser zoom or protocol state.
- The issue-71 retro presentation pass is intentionally symbolic, not sprite-driven: keep simple geometric floor diamonds, paint overlays, and player markers; preserve the limited green/gold/dark palette and monospace bitmap-style typography across Pixi labels and DOM chrome when changing those surfaces.
- Painted tile art is presentation-only and keyed from the existing server palette values: `0xE6194B` lava, `0x3CB44B` grass, `0xFFE119` sand, `0x4363D8` water, `0xF58231` copper, `0x911EB4` crystal, `0x46F0F0` ice, `0xF032E6` flowers. Keep unknown or missing assets playable via the fallback diamond; do not change paint color count or protocol to add tile graphics.
- ISO character sprites are generated as the four DIAGONAL (ordinal) facings — `north-east`, `south-east`, `south-west`, `north-west` — via PixelLab's 8-direction endpoint (`web/tools/contract.mjs`, driven by `gen-asset.mjs --facings ordinal`). Those are the facings that read correctly under the iso camera; cardinal side/front/back views look wrong here. The manifest keys idle frames and any walk-cycle frame sequences by those ordinal names; do not register ordinal walk art under cardinal keys. Do not bake ground shadows into character art — `render.ts` grounds the sprite itself (auto-detected feet row) and removes the procedural token shadow, so a baked shadow only doubles up and reads as hovering (issue #92).
- WASD moves along world axes, which look diagonal under iso (acceptable for MVP).
- the production engine URL comes from Vercel's `WS_URL` env via `/config.json`; there is no client rebuild on URL change. `web/api/config.js` + `web/vercel.json` wire this up — see `docs/deploy.md`.
